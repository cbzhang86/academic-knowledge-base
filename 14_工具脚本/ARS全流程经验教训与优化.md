## 二、运行中发现的问题（按严重程度排序）

### （一）管线前环境问题

#### 问题 1：Agent API 400 错误 — `thinking:disabled + reasoning_effort` 冲突
**严重程度**：管线阻塞
**表现**：Claude Code v2.1.166+ 为子 Agent 注入 `thinking:{type:disabled}` + 会话级 `reasoning_effort`，DeepSeek API 拒绝该组合，返回 400。
**根因**：Claude Code 框架行为变更（GitHub Issue #1397）。
**解决方案**：回退至 v2.1.165。
**经验**：使用第三方 API 代理时应锁定 Claude Code 版本，升级前验证兼容性。

#### 问题 2：ARS 命令模型设置导致输出缩水
**严重程度**：严重（字数从 13K 降至 2K）
**表现**：将 ARS 命令文件的 model 从 sonnet/opus 改为 haiku，Agent 输出能力和上下文窗口严重受限。
**根因**：调试代理 400 错误时误修改了 ARS 配置文件。
**解决方案**：恢复 ARS 文件至原始模型设置。
**修复命令**：
```bash
cd /c/Users/Administrator/.claude/plugins/cache/academic-research-skills/academic-research-skills/3.12.0/commands
git checkout -- *.md    # 恢复命令文件的模型设置
git -C ../deep-research/agents checkout -- *.md  # 恢复代理文件
```

### （二）管线运行中的内容质量问题

#### 问题 3：参考文献 [9][10] 为幽灵引用
**严重程度**：CRITICAL（Stage 4.5 检出）
**表现**：Fourcade & Healy (2017) 和 Zuboff (2019) 出现在参考文献列表，但正文全文从未引用。
**根因**：在修订期间新增了多个新文献（DiMaggio、Winner 等），但未删减原有的装饰性文献。
**预防**：每个修订轮次后要做一次"正文引用↔文献列表"的幽灵引用检查。

#### 问题 4：正文引用无对应文献条目
**严重程度**：HIGH（Stage 4.5 检出）
**表现**：日本厚生劳动省(2023)、欧盟《欧洲无障碍法案》(EAA)、欧盟《数据治理法案》(DGA) 在正文中被引用，但参考文献列表中无对应条目。
**根因**：修订时引入了新的国际比较内容，但未同步更新参考文献。
**预防**：修订时每条新增引用必须立即添加对应文献条目，不能留到"最后统一处理"。

#### 问题 5：参考文献[5]页码在修订稿中未被修正
**严重程度**：HIGH（Stage 2.5→4.5 跨阶段遗漏）
**表现**：Stage 2.5 已发现青连斌等页码应为 38-46 而非 1-15，但 Stage 4 修订后仍未修复。
**根因**：Stage 2.5→Stage 4 的修复清单未建立闭环追踪机制。
**预防**：建立"问题追踪清单"，每个问题标注 P0-P3 优先级和"已修复/待修复"状态，跨阶段传递。

#### 问题 6：R&R 答复信与实际修订不一致
**严重程度**：MEDIUM
**表现**：答复信承诺的"隐性排斥操作化定义"未实际写入正文，国际文献描述位置不准确。
**根因**：R&R 信和修订稿由同一 Agent 生成，但缺少产出后的一致性检查步骤。
**预防**：R&R 信生成后自动验证（grep 检查承诺的关键文本是否在修订稿中出现）。

#### 问题 7：无来源统计数据
**严重程度**：MEDIUM
**表现**："42% 慢性病管理精准度"无任何引用标注。
**根因**：初稿时的 AI 生成内容未被完整性审核在早期发现（Stage 2.5 未覆盖此条）。
**预防**：完整性审核的 Phase C（统计数据验证）应扩展到 100%（而非 30% 抽检）。

#### 问题 8：正文中的框架示意图使用 ASCII 符号
**严重程度**：外观问题
**表现**：`┌──┐` 画框，不符合中文学术论文规范。
**根因**：初稿由 AI 生成时默认使用了代码块模式画图。
**解决方案**：改为规范的三线表（Markdown 表格），python-docx 生成 Word 原生表格。

### （三）管线自身的缺失

#### 问题 9：Stage 5 无中文学术格式输出能力
**严重程度**：功能缺失
**表现**：Pandoc 转换的 .docx 无中文字体、无双栏、无规范页边距。
**解决方案**：开发 `create_formatted_docx.py`，支持 5 种格式模板，注册为 Skill `report/format_cn`。
**经验**：ARS 的学术论文格式支持主要为英文设计，中文学术论文需要额外的格式后处理步骤。

#### 问题 10：Stage 3' 结果呈现不完整
**严重程度**：流程不畅
**表现**：Agent 输出了完整的 3 份审稿报告+决策函，但助手只展示了汇总表，用户需要追问才看到完整内容。
**根因**：助手在传递 Agent 输出时做了过度简化。
**教训**：Agent 输出的完整审稿报告必须全部呈现，不得仅展示汇总摘要。

## 三、系统性优化措施

### 优化 1：建立"问题追踪清单"跨阶段传递机制

在每个阶段结束时，生成一份结构化的状态清单，传递给下一阶段：

```yaml
# 问题追踪清单格式
阶段: Stage X
问题数: N
问题列表:
  - id: "I-001"
    描述: "..."
    严重级别: CRITICAL/HIGH/MEDIUM/LOW
    状态: OPEN/FIXED/OVERRIDDEN
    首次发现: Stage 2.5
    最后一次检查: Stage 4.5
    修复验证: PASS
```

将该清单写入 `10_研究输出/定稿/问题追踪清单.yaml`，当管线在各阶段推进时，作为中间产物传递。

### 优化 2：Stage 4 修订后增加"前后一致性自检"

在 Stage 4 修订完成、Stage 3' 再审之前，自动运行以下检查：

```bash
# 1. 检查 R&R 答复信承诺的修改是否在修订稿中存在
grep -f 承诺的关键词列表 修订稿.md

# 2. 幽灵引用检查
grep -oP '\[\d+\]' 参考文献 | sort -u  # 文献列表序号
grep -oP '（[^）]*\[\d+\]）' 正文      # 正文引用

# 3. 参考文献条目匹配
# 检查正文引用的所有（作者，年份）是否在文献列表中有对应

# 4. 统计声明来源
# 提取所有含数字的句子，检查是否有引用标注
grep -oP '[^。]*[0-9]+%?[^。]*。' 正文 | grep -v '（'
```

### 优化 3：创建论文写作前 Checklist

在启动 ARS 管线前，执行以下清单：

- [ ] 确认 Claude Code 版本兼容：`claude -V` → 2.1.165（非 2.1.166+）
- [ ] 确认 ARS 命令文件模型设置未被修改
- [ ] 确认论文初稿至少 5,000 中文字（否则 Stage 3 审稿无意义）
- [ ] 确认参考文献至少 8 条
- [ ] 确认所有统计声明有引用标注
- [ ] 确认 .docx 格式后处理脚本可用

### 优化 4：ARS 全流程进度看板

在 `14_工具脚本/ARS进度看板.md` 中维护运行状态，每完成一个阶段自动写入。

### 优化 5：中文学术论文格式后处理自动化

将 `create_formatted_docx.py` 嵌入 ARS Stage 5 的**后置步骤**中。

当前方案（手动）：
```bash
python 14_工具脚本/report/create_formatted_docx.py convert \
  -i 终稿.md -o 终稿_学报版.docx --style 学报
```

目标方案（纳入管线）：
在 Stage 5 完成后自动调用 `report/format_cn`，根据用户选择或配置文件自动应用对应模板。
