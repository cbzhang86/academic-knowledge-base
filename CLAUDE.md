## 一、项目概述

面向公共管理学、社会学领域的**学术文献管理 + 系统综述工作流知识库**。核心目标：
- 系统追踪 5 个研究方向（公共管理学、社会学、老龄化研究、青少年研究、交叉研究）的 CSSCI 论文
- 从**检索→精读→跨篇沉淀**，形成可复用的知识资产
- **支持论文选题、开题报告、系统综述写作**

**当前规模**：115 篇精读报告（全部 v10 完整格式）+ 115 篇PDF原文 + 跨篇沉淀素材覆盖126篇 + 9 篇方法笔记（2026-06-10）
**论文产出**：《技术在场、服务缺席》（13,135字）— 已通过 ARS 全流程管线（审稿→修订→定稿）
**版本**：v10.5 — 全量数据对齐完成（PDF数==精读数==索引数==115）

## 四、精读报告标准（v10 完整格式）

每篇精读报告必须包含以下 7 个部分，长度 175-205 行：

1. **YAML frontmatter**（9 字段）：`title/tags/created/theme/study_area/data_source/methodology/core_variable/key_finding/relevance`
2. **论文信息表**：标题/作者/期刊/DOI/基金/优先级/入库路径/MD5/CSSCI 等
3. **研究概述**：一句话定位（≤30字）+ 研究对象/核心问题/研究情境 + 理论框架 + 研究方法表
4. **质量评分（7 维）**：研究问题/理论框架/设计/证据/深度/写作/创新 — **每维必须有评价性理由**
5. **内容拆解**：核心发现 + 关键论点（原文引用+页码）+ 核心概念定义 + 核心公式/指标 + 创新贡献 + 不足与展望
6. **我的思考**（4 子项必须全部存在）：最有启发的点 / 可借鉴之处 / 待验证/存疑点 / 与我的研究关联
7. **标签表 + 元信息**

> ⚠️ **铁律**：不准简写 / 不准跳过审核（每篇提交 Explore 子智能体审核）/ 直接引文必须逐字匹配原文
> ⚠️ **铁律2**：**入库路径行必须严格按照 `| **入库路径** | \`01_论文原文/方向/文件名.pdf\` |` 完整格式**，缺失前缀或反引号不匹配会导致校验失败
> ⚠️ **铁律3**：**报告存放目录必须与PDF目录一致**（PDF在哪方向，精读报告就在哪方向），不可跨目录存放
> ⚠️ **铁律4**：**文件名必须含年份_作者前缀**（如 `2026_张三_论文标题_精读报告.md`），避免同名重复

## 五、核心工作流

工作流由 **5 个阶段 + 必选自检** 组成。**每一阶段完成后必须执行自检**，不满足条件不进入下一阶段。

### Agent 角色映射（谁在什么阶段做什么）

| 阶段 | 负责 Agent | 调用的 Skill | 产出 |
|:----:|:-----------|:-------------|:-----|
| **0-1** | **采集Agent** | `collect/search` → `metadata/dedup` → `collect/download` → `metadata/archive` | PDF 文件 + dedup 登记 |
| **2** | **精读Agent** | `review/draft`（按模板写作）→ `review/check`（自查） | v10 精读报告 |
| **3** | **综合Agent** | `extract/regenerate` → `report/daily` | 写作素材 + 日报 |
| **维护** | **运维Agent** | `tools/sync` → `metadata/dedup` → 更新看板/日志 | Obsidian 同步 + 状态对齐 |
| **写作** | **审稿Agent** | `/ars-full` → `report/format_cn` | 论文定稿 |

### 阶段 0：检索与采集
1. 从关键词池（固化在 `14_工具脚本/采集/ncpssd_cdp.py` 的 `KEYWORD_POOL` 中）选关键词
2. 启动 Edge CDP → `python "14_工具脚本/采集/ncpssd_cdp.py" download "关键词"`
3. 或批量：`python "14_工具脚本/采集/ncpssd_cdp.py" download-pool "方向名"`
4. 记录到 `14_工具脚本/检索日志.md`
> ✅ **自检**：PDF 魔数校验（`%PDF` 开头）+ 文件名标准化（`年份_作者_标题.pdf`）+ 不相关论文已过滤

### 阶段 1：下载与入库
> **操作前必须读**：`memory/ncpssd-download-pipeline.md` + `memory/critical-db-rules.md`

1. MCP前置验证（done.sqlite 四项一致）
2. 双壳去重（Zotero DB + 01_论文原文）
3. 搜索 → 左键单击下载（**Input.dispatchMouseEvent**，禁用 .click()）
4. 移动 PDF 到 `01_论文原文/{方向}/`
5. 关 Zotero → `python 14_工具脚本/入库/batch_import_zotero.py --append` → 四项验证 → 更新 done.sqlite → 开 Zotero
6. 更新 `14_工具脚本/进度看板.md`
> ✅ **自检**：`python 14_工具脚本/metadata/dedup.py rebuild && python 14_工具脚本/metadata/dedup.py status` → 三数一致（PDF 数 == dedup 索引数 == Zotero 论文数）

### 阶段 2：精读分析
1. 按 `analysis_template.md` 逐篇写（175-205 行，含 7 维评分+我的思考）
2. 提交 Explore 子智能体审核 → 修正 → 下一篇
> ✅ **自检**：精读报告文件名必须为 `*_精读报告.md`；`created` 字段日期为今天；入库路径指向真实 PDF
> 🧠 **经验教训**：Agent 写入精读报告时，**入库路径（`入库路径`）行必须以完整格式 `| **入库路径** | \`01_论文原文/方向/文件名.pdf\` |` 出现**，否则无法被索引。已批量修复 8 篇此类错误。
> 🧠 **方向一致性铁律**：**精读报告存放目录必须与PDF所在目录一致**。PDF在`01_论文原文/社会学/`，精读报告就必须在`02_精读报告/社会学/`。已批量清理 14 篇跨方向报告。

### 阶段 3：批量写作（新增 — 适合补缺口场景）
当需要一次性补写多篇精读报告时（如新采集一批PDF），使用 Workflow 并行 Agent：

```bash
# Workflow 并行写作（自动处理 PDF 读取 + 报告生成 + 文件写入）
# 参考：本日批量写入了 19 篇精读报告（交叉7+老龄1+公共管理6+社会5）
```

> ✅ **自检**：每篇报告行数 175-205；入库路径指向真实 PDF；文件名含年份_作者前缀
> ✅ **目录对齐**：批量报告写入后，必须手动检查**报告存放目录是否与PDF目录一致**

### 阶段 4：跨篇沉淀
1. 一键重跑：`python "14_工具脚本/提取/regenerate_all.py"` → 检查摘要/理论/空白篇数与精读报告数匹配
2. 生成日报：`python "14_工具脚本/报告/daily_weekly_monthly.py" daily` → 确认今日篇数正确
3. 生成周报：`python "14_工具脚本/报告/daily_weekly_monthly.py" weekly`
4. 同步 Obsidian：`python "14_工具脚本/tools/sync.py"`
> ✅ **自检**：`cat 05_报告/日报/日报_YYYY-MM-DD.md` → 今日精读数 = 今天写的报告数，评分已显示，我的思考已提取

### 阶段 5：维护与对齐
1. 更新 `14_工具脚本/进度看板.md` → 数据与实际对齐
2. 更新 `14_工具脚本/检索日志.md` → 记录本次采集详情
3. 验证全链一致性：
   - `python 14_工具脚本/提取/regenerate_all.py` → 跨篇沉淀重跑
   - `python 14_工具脚本/报告/daily_weekly_monthly.py daily` → 日报更新
   - `python 14_工具脚本/tools/sync.py` → Obsidian 同步
   - **硬校验**：各方向 PDF 数 == 精读报告数 == dedup 索引数
> ✅ **自检**：三数对齐 + 四项验证通过后结束本轮工作流

### 阶段 6：推送前拦截关卡（PUSH 前自动执行）

**此阶段通过 git pre-push hook 自动强制执行**，无需手动触发。

```bash

```

**自动检查 6 项**：

6. （framework 推前）-> 框架分支的 README/CLAUDE 版本同步

> 任意一项不通过 -> git push 被拦截 -> 修复后重推
> 如需紧急推送（不推荐），使用 `git push --no-verify` 跳过

**安装 hook**：
```bash

```

### 阶段 7：论文写作（ARS 全流程）

本知识库采用 **知识注入管线（Knowledge Injection Pipeline）** 模式，将跨篇沉淀直接嵌入 ARS 论文写作 Agent 的上下文，实现"知识库不只为人类阅读，也为 AI 写作时可用"。

```
03_学术写作素材库/                    ← 精读沉淀（理论·论点·空白·方法）
     │  ars_kb_bridge.py 按阶段切分
     ▼
10_研究输出/ars_output/kb_pack_*     ← 阶段知识包（人+AI可读）
     │  和本阶段输出一起 ❝打包❞ 给下阶段
     ▼
ARS 各阶段 Agent ← 审稿/修订/定稿时拥有全部知识库背景
     │
     └→ 确保：审稿不遗漏关键理论、修订能找到支撑文献、定稿不遗漏重要引文
```

**核心流程**：你给我初稿 → 我分析论文方向 + 提取引文 → 按需逐段生成知识包 → 每步的包与当前产出一起传给下一步。

### ⚠️ 归档铁律：完整保存，不准摘要

每阶段完成后，对话中产生的完整报告**必须全文写入文件**，不得写摘要版。标准如下：

| 阶段 | 文件 | 最低要求 |
|:-----|:-----|:---------|
| 02_完整性审核 | `完整性审核报告.md` | >2,000字，逐条列出问题+位置+建议 |
| 03_审稿报告 | `审稿报告.md` | >3,000字，每位审稿人展开评价+具体扣分点 |
| 04_修订稿 | `修订稿.md` | 实际修订正文，非修订说明摘要 |
| 05_最终完整性审核 | `最终完整性报告.md` | >1,500字，逐条三段式（要求→修订→验证）|

归档位置：`10_研究输出/_papers/{论文名}/{阶段目录}/`
知识包注入脚本：`ars_kb_bridge.py --stage N --topic X --paper Y`

**技术在场服务缺席 部分报告仅存摘要版（对话已过窗口无法回溯），后续论文不得出现此情况。**

### ⚠️ 关键经验教训（2026-06-10）

#### 教训1：ARS skill 加载失败时的降级方案

ARS 通过 Skill 工具动态加载，**compact（上下文压缩）后注册信息会丢失**。再次调用 `/ars-full` 可能失败。

**正确做法**：
- 第一次失败后，应**重新调用 Skill 工具重试**，而非自行降级
- 如果 Skill 仍然不可用，必须用 **Agent 工具启动独立子智能体** 来模拟多 Agent 结构
- **严禁**在没有独立验证的情况下，由同一 Agent 同时担任审稿人、修订者和验证者

| 角色 | 责任人 | 独立性要求 |
|:-----|:-------|:----------|
| 审稿 | Agent A（或 Workflow 多 Agent） | 独立于修订者 |
| 修订 | I（主对话） | — |
| 验证 | Agent B（新的子智能体，不继承上下文） | 从零开始，不信任之前输出 |

#### 教训2：统计量必须有真实来源

修订时补充审稿人要求的数据（如 Cohen's Kappa），必须：
- 有实际计算过程或明确来源
- 或在修订稿中注明"建议补充"而非呈现为"已计算"
- 验证环节必须由独立 Agent 检查每项统计量的来源真实性

| 阶段 | 名称 | 产出 | 输入（含知识包）|
|:----:|:------|:-----|:------------|
| **2.5** | 完整性审核 | 完整性审核报告 | 论文+引文对照知识包 |
| **3** | 同行评审 | 5 份审稿报告 + 决策函 | 论文+完整性报告+审稿背景包 |
| **4** | 修订 | 修订稿 + R&R 答复信 | 论文+审稿报告+理论推荐包 |
| **3'** | 再审 | 验证性评审报告 | 修订稿+再审知识包 |
| **4.5** | 最终完整性 | 最终完整性报告 | 修订稿+引文审计包 |
| **5** | 定稿 | 终稿 .md | 前面所有产出 |

python 14_工具脚本/tools/sync.py

## 九、配置指南

### 路径配置（config/structure.yaml）
默认中文目录名，用户可自定义：
```yaml
directories:
  01_raw: "01_论文原文"   # 改为 "01_raw_papers"
  02_reports: "02_精读报告" # 改为 "02_reading_reports"
```

### 领域配置（config/areas.yaml）
新增领域只需加一段，不改代码：
```yaml
areas:
  公共管理学:
    keywords: ["数字政府", "基层治理"]
    sources: ["ncpssd", "openalex"]
  计算机科学:      # 新增领域
    keywords: ["deep learning", "transformer"]
    sources: ["arxiv", "openalex"]
```

### 归档后端（config/metadata.yaml）
- `filesystem`：默认，零依赖，clone 即用
- `zotero`：可选，需 Zotero 桌面端
- 首次运行 `python global_config.py init` 交互式选择
